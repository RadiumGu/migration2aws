# Analytics & BI Services Design Rubric

**Applies to**: Azure Synapse Analytics, Data Factory, Stream Analytics, HDInsight, Power BI, Azure Data Explorer (Kusto), Azure Databricks.

**Quick lookup**: No fast-path — analytics migrations are predominantly Medium-High complexity and require rubric evaluation. The one exception is Databricks on AWS (direct migration retained).

## 2. Eliminators (Hard Blockers)

| Source | AWS Candidate | Blocker |
|--------|---------------|---------|
| Synapse with T-SQL MERGE statements using Synapse-specific extensions | Redshift | Redshift MERGE (preview) or rewrite as INSERT+UPDATE |
| Synapse with PolyBase external tables pointing to ADLS | Redshift | Use Redshift Spectrum with Glue Catalog (S3 as data source) |
| Data Factory with on-prem SSIS packages hosted in Azure-SSIS IR | Glue | Rewrite in PySpark or use SSIS-on-EC2; Glue has no SSIS compat |
| Stream Analytics with > 192 SU (Streaming Units) | Kinesis Flink | Re-architect; scale limits differ |
| Power BI with Row-Level Security via AD groups + DAX | QuickSight | QuickSight RLS via rules file / AWS IAM; complete rebuild |
| Kusto with cross-cluster queries | OpenSearch | No cross-cluster equivalent natively; use cross-cluster search or federate via Glue |
| Databricks Delta Lake with Azure Data Lake direct path | Databricks on AWS | Delta format portable; storage path changes (abfss:// → s3://) |
| HDInsight Storm topologies | EMR | Storm not supported in EMR 7.x; rewrite to Flink or Spark Streaming |

## 3. Signals (Decision Criteria)

### Synapse Analytics → Redshift / Athena / EMR (High Complexity)

Synapse is a multi-service platform. Map by sub-service:

| Synapse Component | AWS Target | Notes |
|-------------------|------------|-------|
| Dedicated SQL Pool (DWU-based) | **Redshift Provisioned** | SQL dialect: T-SQL → Redshift PostgreSQL. AWS SCT helps. |
| Serverless SQL Pool | **Redshift Serverless** OR **Athena** | Redshift Serverless for complex SQL; Athena for ad-hoc on S3 |
| Spark Pool | **EMR Serverless** or **Glue** | PySpark reusable; job config rewritten |
| Pipelines | **Glue Workflows** or **Step Functions** | Pipelines are Data Factory under the hood |
| Synapse Studio | QuickSight + Athena + Glue Studio | No unified IDE — split across tools |

**SQL dialect differences** (T-SQL → Redshift):
- `TOP N` → `LIMIT N`
- `GETDATE()` → `CURRENT_DATE`
- `CHARINDEX` → `POSITION` / `STRPOS`
- `IDENTITY(1,1)` → `IDENTITY(0,1)` or `GENERATED BY DEFAULT AS IDENTITY`
- Stored procedures: T-SQL → PL/pgSQL (AWS SCT translates most)

**Flag**: Always add to `special_mappings[]` with risk=high, notes: "SCT assessment required; SQL dialect + pipeline rebuild; 8-16 weeks"

### Data Factory → Glue (High Complexity)

- **Pipelines** → Glue Workflows (or Step Functions for complex orchestration)
- **Copy Data activity** → Glue job with DynamicFrame or Spark DataFrame
- **Data flow (mapping)** → Glue Studio visual editor (equivalent concept)
- **Triggers (schedule, tumbling window)** → EventBridge Scheduler / EventBridge rules
- **Linked Services (connections)** → Glue Connections (JDBC, S3, Kinesis)
- **Integration Runtimes** → Glue runs on AWS-managed compute (no IR concept)

**Effort**: 2-4 weeks per moderate pipeline (~50 activities). Rewrite in PySpark.

### Stream Analytics → Kinesis Data Streams + Managed Service for Apache Flink (Medium)

- **Input: Event Hub** → Kinesis Data Streams
- **Input: IoT Hub** → IoT Core + Kinesis
- **Input: Blob Storage (reference data)** → S3 + DynamoDB (as ref data source for Flink)
- **Query (ASA SQL)** → Flink SQL or Kinesis Data Analytics Studio (Apache Zeppelin)
- **Output: SQL DB, Power BI, Cosmos, etc.** → Map per sink (RDS, QuickSight, DynamoDB)
- **Windowing**: Tumbling/Hopping/Sliding → Flink windowing (equivalent)

**ASA SQL → Flink SQL translation**: Most constructs (window, join, aggregation) translate directly. Late-arriving events and out-of-order semantics differ — review watermarking.

### HDInsight → EMR (Medium)

- **HDInsight Hadoop/Spark** → **EMR** (Spark, Hive, Presto, HBase)
- **HDInsight Kafka** → **MSK** (managed Kafka) — prefer over running Kafka on EMR
- **HDInsight HBase** → **EMR HBase on S3** (or DynamoDB if pattern fits KV)
- **HDInsight Storm** → EMR does not include Storm; rewrite to Flink or Spark Streaming
- **Cluster configurations**: Bootstrap scripts reusable with adaptations; storage (WASB/ADLS → s3://)
- **Ranger / ACL** → **Lake Formation** fine-grained access control

### Power BI → QuickSight (High)

- **Workspaces** → QuickSight folders + sharing
- **Datasets** → QuickSight datasets (re-import from source)
- **Reports** → QuickSight analyses (manual rebuild — no conversion tool)
- **Dashboards** → QuickSight dashboards
- **DAX measures** → QuickSight calculated fields (different syntax)
- **Row-Level Security (RLS)** → QuickSight RLS rules file
- **Paginated reports (SSRS-style)** → QuickSight paginated reports
- **Power BI Embedded** → QuickSight Embedded Analytics
- **Composite models / DirectQuery** → QuickSight SPICE + direct query modes

**Effort**: 4-12 weeks for 50+ reports; no automated conversion exists.

### Azure Data Explorer (Kusto) → OpenSearch (High)

- **Kusto Query Language (KQL)** → **OpenSearch Query DSL** (JSON) or **SQL plugin**
  - KQL's pipe syntax has no direct equivalent; queries fully rewritten
- **Tables** → **OpenSearch indices**
- **Update policies, materialized views** → OpenSearch rollups + ingest pipelines
- **Ingestion (Event Hub → ADX)** → Kinesis Firehose → OpenSearch OR Logstash
- **Time-series only workloads** → Consider **Amazon Timestream** (better fit for pure TS)

### Azure Databricks → Databricks on AWS (Preferred, Low Complexity)

- **Databricks on AWS is native** — workspace portable, notebooks/jobs directly runnable
- Storage path: `abfss://container@account.dfs.core.windows.net/path` → `s3://bucket/path`
- Unity Catalog: Azure-backed → AWS-backed metastore
- Delta tables: Format identical, just relocate storage
- MLflow: Portable
- **If not keeping Databricks** → EMR (Spark + Hive) or Glue (Spark-based ETL)

**Effort**:
- Databricks on AWS: **1-3 weeks** (Low complexity)
- Switch to EMR/Glue: **6-12 weeks** (High complexity)

## 4. 6-Criteria Rubric

Apply in order; first match wins.

1. **Eliminators**: Blocking feature? Switch candidate.
2. **Operational Model**: Managed (Redshift Serverless, Glue, Kinesis) vs self-managed (EMR with Spot, Kafka self-hosted).
   - Prefer managed unless cost-critical large-scale workload
3. **User Preference**:
   - `q2 = cost` → Redshift Serverless / Athena / EMR Spot / Glue (pay-per-use)
   - `q2 = capability` → EMR with full Hadoop ecosystem; Redshift Provisioned with RA3
   - `q5 = analytics` (from clarify) → directly routes here
4. **Feature Parity**: SQL dialect, query language, visualization?
   - Synapse T-SQL vs Redshift PostgreSQL: SCT helps
   - Power BI DAX vs QuickSight calculated fields: manual translation
5. **Cluster Context**: Existing data on S3? Glue Catalog populated? Prefer services using same catalog.
6. **Simplicity**: Fewer moving parts. Redshift Serverless > Redshift Provisioned + Spectrum + Glue Catalog (unless need ad-hoc S3 query).

## 5. Decision Tree

```
Is it Synapse?
├─ Dedicated SQL Pool → Redshift Provisioned [+ SCT; T-SQL → PostgreSQL]
├─ Serverless SQL → Redshift Serverless OR Athena
├─ Spark Pool → EMR Serverless / Glue
└─ Pipelines → Glue Workflows / Step Functions

Is it Data Factory?
└─ → Glue [+ pipeline rewrite in PySpark; 2-4 wk/pipeline]

Is it Stream Analytics?
└─ → Kinesis Data Streams + Managed Flink [+ ASA SQL → Flink SQL translation]

Is it HDInsight?
├─ Hadoop/Spark/Hive/Presto → EMR
├─ Kafka → MSK (managed)
├─ HBase → EMR HBase on S3
└─ Storm → rewrite to Flink

Is it Power BI?
└─ → QuickSight [+ full report rebuild; 4-12 wk for 50+ reports]

Is it Kusto (ADX)?
├─ Logs/search → OpenSearch [+ KQL → OpenSearch DSL]
└─ Pure time-series → Timestream

Is it Databricks?
├─ Keep on Databricks → Databricks on AWS [direct migration, 1-3 wk]
└─ Switch → EMR / Glue [+ 6-12 wk rewrite]
```

## 6. Examples

### Example 1: Synapse Dedicated SQL Pool (DW1000c, 10 TB)

- Azure: `Microsoft.Synapse/workspaces/sqlPools`, DW1000c, 10 TB data, 250 stored procedures
- Pass 2 rubric:
  - Eliminators: No PolyBase to ADLS; no CLR assemblies → PASS
  - Operational Model: Redshift Provisioned (managed)
  - User Preference: q2=capability → Redshift RA3
  - Feature Parity: T-SQL → Redshift PostgreSQL via SCT
- → **Redshift Provisioned `ra3.4xlarge × 2 nodes`**, Redshift Spectrum for S3 reference data
- special_mappings: add with risk=high, notes: "SCT run required; 250 stored procs to translate (~8 weeks); data migration via UNLOAD/COPY ~2 weeks"
- Confidence: `inferred`

### Example 2: Data Factory with 80 Pipelines

- Azure: `Microsoft.DataFactory/factories`, 80 pipelines, 6 linked services (ADLS, SQL DB, Cosmos, Event Hub, SFTP, REST API)
- Pass 2 rubric:
  - Eliminators: No Azure-SSIS IR → PASS
  - Operational Model: Glue (managed)
- → **Glue jobs + Glue Workflows** (80 jobs); Step Functions for cross-service orchestration
- special_mappings: add with risk=high, notes: "Rewrite in PySpark; 80 pipelines × 2-4 wk each ≈ 4-6 months; parallelize across team"
- Confidence: `inferred`

### Example 3: Azure Databricks Premium (Q = keep on Databricks)

- Azure: `Microsoft.Databricks/workspaces` Premium, 30 notebooks, 15 jobs, Unity Catalog, Delta Lake in ADLS Gen2
- Pass 1 (fast-path to Databricks on AWS): **Databricks on AWS Premium**
- Data migration: ADLS Gen2 → S3 via AzCopy/DataSync; Delta format preserved
- Unity Catalog: re-create in AWS
- Confidence: `deterministic` (Databricks stays; path is direct)

### Example 4: Power BI Premium with 120 Reports

- Azure: Power BI Premium per-capacity (P2), 120 reports, 40 datasets, RLS on 15 reports, 5 paginated reports
- Pass 2 rubric:
  - Eliminators: RLS via DAX → QuickSight RLS rules file (rebuild)
- → **QuickSight Enterprise**, 40 datasets re-imported, 120 analyses rebuilt, 120 dashboards
- special_mappings: add with risk=high, notes: "Full rebuild; no automation; DAX → QuickSight calculated fields; 10-14 weeks for 120 reports"
- Confidence: `inferred`

### Example 5: Azure Data Explorer (Log Analytics Pattern)

- Azure: `Microsoft.Kusto/clusters`, 2 nodes, 5 TB hot cache, 200 GB/day ingest via Event Hub, 80 saved KQL queries
- Pass 2 rubric:
  - Eliminators: No cross-cluster queries → PASS
  - Operational Model: OpenSearch managed
- → **OpenSearch Service (3 master + 6 data `r6g.xlarge.search`)**, Kinesis Firehose ingestion, 80 queries rewritten to DSL
- special_mappings: add with risk=high, notes: "KQL → OpenSearch DSL rewrite; 80 queries × 2-4h each ≈ 4-6 weeks"
- Confidence: `inferred`

### Example 6: Stream Analytics Job (IoT Telemetry)

- Azure: `Microsoft.StreamAnalytics/streamingjobs`, 6 SU, input Event Hub, output SQL DB, 3 tumbling windows, reference data from Blob
- Pass 2 rubric:
  - Eliminators: 6 SU << 192 SU limit → PASS
  - Operational Model: Managed Flink
- → **Kinesis Data Streams (2 shards)** + **Managed Service for Apache Flink** (equivalent SU); output to RDS; S3 for reference data (joined in Flink)
- special_mappings: add with risk=medium, notes: "ASA SQL → Flink SQL translation 2-3 weeks"
- Confidence: `inferred`
